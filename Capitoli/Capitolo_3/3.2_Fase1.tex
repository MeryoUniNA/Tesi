\section{Fase 1 - FaultOracle e Localizzazione Causale}
\label{sec:fase1}

La prima fase di HybridRepair, denominata \textbf{FaultOracle}, risponde a una
domanda precisa: \textit{dove e perché} il programma difettoso produce un comportamento anomalo.
Il termine ``oracolo'' è mutuato dalla teoria del software testing, dove indica un meccanismo capace
di stabilire la correttezza di un output rispetto alle specifiche attese. FaultOracle opera qui come
\textbf{oracolo diagnostico}: non si limita a rilevare il fallimento, ma emette una rappresentazione
strutturata della radice causale del difetto, consumata dalle fasi successive della pipeline.

Il motore di inferenza sottostante è \textbf{LogicFL} \cite{kim2024identifying}, un sistema di \textit{fault localization} basato
su ragionamento logico-deduttivo, implementato in SWI-Prolog. A differenza dei localizzatori spettro-basati
tradizionali, che producono una lista di righe sospette calcolate su coefficienti di
correlazione tra copertura ed esiti dei test, LogicFL costruisce esplicitamente delle \textit{catene causali}:
tracce formali della propagazione di un valore anomalo (un riferimento \texttt{null}) dalla sorgente
originaria fino al crash a runtime. Il processo deduttivo opera su una \textit{knowledge base} di fatti
Prolog, derivati dall'analisi statica della struttura del codice e dall'analisi dinamica dell'esecuzione dei test.

\textbf{Esempio Guida: Il Bug Csv-4} \\
Per rendere concreti i concetti astratti di FaultOracle, l'intera trattazione seguente utilizzerà come \textit{running example} il bug \textbf{Csv-4} dal dataset Defects4J. Questo difetto causa una \texttt{NullPointerException} nel progetto Apache Commons CSV quando si tenta di costruire una mappa a partire da header mai inizializzati. Le quattro righe che ci interessano sono:

\begin{lstlisting}[language=Java]
// org.apache.commons.csv.CSVParser

// Nel costruttore: il campo viene riempito col risultato del metodo
this.headerMap = this.initializeHeader();    // ASSEGNAZIONE (Riga 244)

public Map<String, Integer> getHeaderMap() {
    // CRASH (Riga 288): this.headerMap e' null
    return new LinkedHashMap<String, Integer>(this.headerMap); 
}

private Map<String, Integer> initializeHeader() throws IOException {
    Map<String, Integer> hdrMap = null; // ORIGINE (Riga 325)
    /* ... se non ci sono header definiti, hdrMap resta null ... */
    return hdrMap; // PROPAGAZIONE (Riga 351)
}
\end{lstlisting}

Prima di qualunque formalismo, proviamo a ragionare come farebbe uno sviluppatore, \emph{all'indietro} a
partire dal crash: il programma si interrompe a riga~288 perché \texttt{headerMap} è \texttt{null}; ma
\texttt{headerMap} era stato riempito a riga~244 col risultato di \texttt{initializeHeader()}; quel metodo, a
riga~351, restituisce \texttt{hdrMap}; e \texttt{hdrMap} era stato posto a \texttt{null} a riga~325.
\textbf{Quella è l'origine.} Tutto ciò che LogicFL fa è automatizzare \emph{esattamente} questa risalita: i
fatti e le regole Prolog delle pagine seguenti non sono che la codifica formale di questo ragionamento.

\subsection{Il Motore LogicFL: Analisi Statica e Dinamica}

\subsubsection{Architettura a tre componenti}
L'architettura di LogicFL è modulare e si articola in tre componenti che operano in sequenza. Ciascun
modulo è una classe Java del package \texttt{logicfl.analyzer.*}, invocabile tramite un corrispondente script shell.
\begin{table}[H]
\centering
\small
\begin{tabularx}{\textwidth}{@{}l l l >{\raggedright\arraybackslash}X@{}}
\toprule
\textbf{Componente} & \textbf{Script} & \textbf{Classe Java} & \textbf{Output principale} \\
\midrule
\textbf{StaticAnalyzer} & \texttt{static\_analyzer.sh} & \texttt{StaticAnalyzer} & \texttt{logic-fl.pl}, \texttt{code-facts.pl} \\
\textbf{DynamicAnalyzer} & \texttt{dynamic\_analyzer.sh} & \texttt{DynamicAnalyzer} & Arricchimento con \texttt{val/3} \\
\textbf{FaultLocalizer} & \texttt{localizer.sh} & \texttt{FaultLocalizer} & \texttt{root-cause.txt}, \texttt{fault\_locs.txt} \\
\bottomrule
\end{tabularx}
\end{table}

Il \texttt{FaultLocalizer} gira su SWI-Prolog, istanziato dentro la JVM tramite il bridge JPL
(\textit{Java--Prolog Library}).\footnote{Concretamente, la JVM carica la libreria nativa \texttt{libjpl.so}
(direttiva \texttt{-Djava.library.path}), che apre un canale bidirezionale tra Java e SWI-Prolog.} Attraverso
questo ponte la classe Java carica i file Prolog, vi asserisce dinamicamente nuovi fatti e valuta le query,
raccogliendone le soluzioni in strutture dati native.

\subsubsection{Analisi Statica: la Generazione dei Fatti Strutturali}

Il primo componente, lo \texttt{StaticAnalyzer}, riceve il codice sorgente Java e ne costruisce una
rappresentazione formale dell'Abstract Syntax Tree (AST), codificata come fatti Prolog. Il programma non
viene eseguito: la sua struttura sintattica e relazionale è tradotta in clausole della logica del primo ordine
(\textit{code instrumentation} statica). Il risultato sono i file \texttt{logic-fl.pl} e \texttt{code-facts.pl}, base
strutturale della \textit{knowledge base}. Il vocabolario dei fatti copre gli elementi della semantica Java
rilevanti per tracciare la propagazione dei valori \texttt{null}; la Tabella~\ref{tab:fatti_prolog} ne riassume
i blocchi fondamentali.

Per dare concretezza al vocabolario della tabella, ecco come l'analisi statica codifica l'assegnazione e l'uso
del campo \texttt{headerMap} in Csv-4:

\begin{lstlisting}[language=Prolog]
assign(f_header_map_69, csvparser_1_expr20, line(csvparser_1, 244)).
return(csvparser_1_expr23, m_get_header_map_61, line(csvparser_1, 288)).
argument(f_header_map_69, 1, csvparser_1_expr23).
\end{lstlisting}

Argomenti come \texttt{f\_header\_map\_69} o \texttt{csvparser\_1\_expr20} sono identificatori interni opachi
generati dall'analizzatore: un prefisso ne segnala il tipo (\texttt{f\_} per i campi, \texttt{m\_} per i metodi,
il suffisso \texttt{\_expr} per le espressioni) e un numero finale disambigua le occorrenze. Nel nostro caso
\texttt{f\_header\_map\_69} è il campo \texttt{headerMap}, \texttt{csvparser\_1\_expr20} è l'espressione
\texttt{this.initializeHeader()} che lo inizializza e \texttt{m\_get\_header\_map\_61} è il metodo
\texttt{getHeaderMap}; ricondurre questi simboli ai rispettivi nomi Java leggibili sarà compito del
\emph{Semantic Grounding} (Sezione~\ref{sec:grounding}). Ricorre inoltre, in quasi ogni fatto, il termine
posizionale \texttt{line(ClassId, LineNum)}, che àncora l'evento a una precisa coppia (classe, riga): un
riferimento univoco all'interno di un progetto multi-file, indispensabile per discriminare omonimie tra classi
differenti.

Per il resto, questi fatti descrivono soltanto la \emph{struttura} del codice: stabiliscono che
\texttt{headerMap} viene assegnato a riga~244 e usato a riga~288, ma non dicono nulla sul fatto che a runtime
valga davvero \texttt{null}. Quell'informazione arriva dall'analisi dinamica descritta di seguito; è solo la
combinazione dei due tipi di fatto a fondare la deduzione del difetto.

\subsubsection{Analisi Dinamica: la Generazione dei Fatti di Valore}

L'analisi statica descrive la struttura del codice ma, per costruzione, non conosce i \textit{valori concreti}
che le variabili assumono durante l'esecuzione. Per asserire che un'espressione valuti effettivamente a
\texttt{null} in un dato punto del flusso, durante il run di un test fallente, serve un'analisi dinamica.

Questo compito è demandato al \texttt{DynamicAnalyzer}, che lavora a livello di \emph{bytecode}: il codice Java
viene compilato normalmente, dopodiché la JVM lo esegue con un agente che inietta istruzioni di osservazione
(senza modificare i file sorgente) nei punti in cui le variabili vengono lette o scritte. Ogni volta che il
test fallente attraversa una di quelle istruzioni, il valore osservato viene registrato. L'esito è l'aggiunta,
in coda a \texttt{logic-fl.pl}, di fatti dinamici della forma \texttt{val(Entità, Valore, Posizione)}: ciascuno
attesta il valore concreto che un'entità ha assunto in un preciso punto dell'esecuzione. Per Csv-4 il fatto
chiave è:

\begin{lstlisting}[language=Prolog]
val(f_header_map_69, null, line(csvparser_1, 288)).
\end{lstlisting}

Generato a tempo di esecuzione durante il test fallente, questo predicato attesta che il campo \texttt{f\_header\_map\_69}
valeva \texttt{null} quando lo stack frame attraversava la riga 288. A differenza dei fatti strutturali,
\texttt{val/3} è una misurazione empirica dello stato del programma. È proprio la combinazione di fatti
strutturali (\texttt{assign/3}, \texttt{ref/3}, \texttt{method\_invoc/3}) e fatti dinamici (\texttt{val/3}) a fondare
la base del ragionatore Prolog: i primi delineano le relazioni possibili tra le entità, i secondi ancorano la
deduzione agli eventi realmente osservati nella riproduzione del fallimento.

Il \texttt{DynamicAnalyzer} formalizza anche lo stack trace catturato al momento del crash:

\begin{lstlisting}[language=Prolog]
test_failure(failure_1, 'org.apache.commons.csv.CSVParserTest', 'testNoHeaderMap').
trace(trace_3, trace_2, m_get_header_map_61, line(csvparser_1, 288), failure_1, target).
\end{lstlisting}

Il predicato \texttt{test\_failure/3} identifica univocamente l'evento di fallimento, esplicitandone classe e
metodo, e funge da ancoraggio radice cui fanno riferimento tutti gli eventi registrati per quel crash. Il
predicato \texttt{trace/6} cattura il singolo frame dello stack, codificando l'ID dell'invocazione, la classe e il
metodo chiamati, la riga sorgente, il contesto di fallimento e un tag binario che marca il frame come
\texttt{target} (codice dell'applicativo) o \texttt{non\_target} (framework di testing come JUnit, o librerie JRE).
Questa categorizzazione è determinante per identificare la linea candidata come vera origine dell'anomalia.


\subsection{Costruzione e Ranking delle Catene Causali}

Il nucleo inferenziale di LogicFL (il file \texttt{npe-rules.pl}) risponde a una domanda precisa: quale entità
del programma è nulla in un dato punto, e da dove proviene tale nullità?. 
Il procedimento si articola in quattro passi: \textbf{(1)} localizzare il sito del crash,
\textbf{(2)} individuare l'espressione nulla, \textbf{(3)} risalire la catena causale fino all'origine,
\textbf{(4)} ordinare le cause per priorità diagnostica. Il punto d'ingresso è il predicato
\texttt{find\_npe\_cause}, che orchestra i quattro passi e restituisce le cause già ordinate.

\subsubsection{Dal crash all'origine: la risalita causale}

I primi tre passi ricostruiscono il cammino del valore \texttt{null}, dal sito del crash fino alla sua origine.
La Figura~\ref{fig:catena_causale} ne mostra il risultato su Csv-4: una catena di quattro anelli, dall'origine
(riga~325) al crash (riga~288).

\textbf{(1) Sito del crash} e \textbf{(2) espressione nulla}. La riga del crash è il primo frame dello stack
trace che appartiene al codice applicativo non a librerie o al framework di test; se il frame più interno è
di libreria, il sistema risale al chiamante (essenziale per le eccezioni lanciate dentro metodi di libreria).
Individuata la riga, LogicFL cerca l'espressione effettivamente nulla. In Csv-4 si tratta di una
\emph{dereferenziazione}: il campo \texttt{headerMap}, attestato come \texttt{null} dai fatti dinamici, viene
usato a riga~288 (questo lo classificherà più avanti come \texttt{UNINITIALIZED\_FIELD}). LogicFL gestisce anche
un secondo caso, non presente in Csv-4, in cui il \texttt{null} non viene dereferenziato direttamente ma
\emph{passato come argomento} a un metodo.

\textbf{(3) Risalita causale.} È il cuore deduttivo di LogicFL,
Il predicato \texttt{cause\_of} prova in ordine tre ipotesi sempre più generali su \emph{dove} sia la causa:

\begin{lstlisting}[language=Prolog]
% Caso 1 - causa immediata al sito del crash
cause_of(npe(Expr, Line), Expr, Line) :- check_val(Expr, null, Line).

% Caso 2 - origine diretta (il null nasce qui)
cause_of(npe(Expr, Line), Cause, Loc) :- originated_from(Expr, Cause, Loc).

% Caso 3 - propagazione transitiva (segui la traccia all'indietro)
cause_of(npe(Expr, Line), Cause, Loc) :- can_be_transferred(Expr, Cause, Loc).
\end{lstlisting}

Il primo caso copre il difetto puramente locale (l'entità è già nulla alla riga del crash); il secondo individua
il punto in cui il \texttt{null} nasce per la prima volta, in Csv-4 la riga~325; il terzo, quello
interessante, segue la propagazione all'indietro di un passo, riconoscendo i tre modi in cui un \texttt{null}
può trasferirsi tra espressioni: per \emph{assegnazione}, per passaggio \emph{parametro\,$\leftarrow$\,argomento}
tra chiamante e chiamato, o tramite \emph{valore di ritorno}. Sono esattamente gli archi della
Figura~\ref{fig:catena_causale}: in Csv-4 il valore di ritorno collega il \texttt{return hdrMap} di riga~351 al
suo chiamante, e l'assegnazione collega il campo a riga~244. Una ricerca in profondità, che tiene traccia dei
nodi già visitati per non entrare in cicli, compone questi singoli passi in catene di lunghezza arbitraria.

\subsubsection{Ranking e output}

L'ultimo passo ordina le cause per rilevanza diagnostica in tre fasi. Il \textbf{Filtering}
scarta le cause formalmente corrette ma inutilizzabili per costruire una patch: le origini interne al codice di
test e i metodi puramente \emph{delegati} (getter o wrapper che si limitano a restituire un parametro ricevuto),
i quali segnalano che l'anomalia risiede più a monte nella catena di invocazioni. La \textbf{Preference} eleva
in cima le cause più azionabili, cioè quelle che soddisfano almeno uno di tre pattern: un metodo con percorsi di
ritorno sia \texttt{null} sia validi (\emph{missing null-return guard}); un'assegnazione di \texttt{null}
avvenuta localmente \emph{prima} della riga del crash; oppure un metodo che è l'unico del codice applicativo
presente nello stack trace. Infine la \textbf{Composizione} antepone le cause preferite alle rimanenti.

L'output è serializzato in due file. Il primo, \texttt{root-cause.txt}, elenca le cause individuate \emph{in
ordine di ranking} (non come cammino lineare): la prima voce coincide con il sito del crash, le successive sono
le restanti cause in ordine di priorità.
\begin{lstlisting}
NPE at line(csvparser_1, 288) / Null Expression - f_header_map_69[headerMap]
	 can be caused by
f_header_map_69[headerMap] - line(csvparser_1, 288).
v_hdr_map_70[hdrMap] - line(csvparser_1, 351).
csvparser_1_literal5[null] - line(csvparser_1, 325).
csvparser_1_expr20['this.initializeHeader()'] - line(csvparser_1, 244).
\end{lstlisting}

Per Csv-4, la riga~351 (\texttt{return hdrMap}) compare subito dopo il sito del crash proprio perché la
Preference la eleva, riconoscendo il pattern \emph{missing null-return guard} in \texttt{initializeHeader()}. In
ogni voce, il campo alfanumerico è l'identificatore interno opaco del motore, mentre il testo tra parentesi
quadre è il referente Java leggibile estratto dall'AST. Questa sequenza ordinata è preservata nel campo
\texttt{causal\_chains} del \texttt{FaultReport}, così che l'LLM possa esaminare le cause a partire dalla più
probabile.

Il secondo file, \texttt{fault\_locs.txt}, fornisce la vista compatta: le sole righe candidate, nello stesso
ordine di priorità.

\begin{lstlisting}
org.apache.commons.csv.CSVParser 288
org.apache.commons.csv.CSVParser 351
org.apache.commons.csv.CSVParser 325
org.apache.commons.csv.CSVParser 244
\end{lstlisting}

\subsection{Semantic Grounding: Dalla Logica Formale all'Identità Java}
\label{sec:grounding}
\subsubsection{Il Problema della Vaghezza Simbolica}
Gli identificatori Prolog opachi (come \texttt{f\_header\_map\_69} o \texttt{csvparser\_1\_expr20}) hanno pieno significato
nel motore di inferenza ma sono privi di significato per un Large Language Model. Questa ``vaghezza simbolica''
fornisce al modello informazioni formalmente corrette ma semanticamente povere, aumentando il rischio di
allucinazioni. Per risolverla, il modulo \texttt{prolog\_grounding.py} applica il \textit{grounding
semantico}: ogni simbolo Prolog viene tradotto nella sua controparte Java concreta (nome, tipo e ruolo
strutturale), fornendo al modello un quadro diagnostico preciso, tipo-corretto e azionabile.

\subsubsection{La Tassonomia degli Identificatori Prolog}
Il primo passo del Semantic Grounding consiste nell'ispezionare l'identificatore Prolog per
classificare il tipo di entità tramite specifiche convenzioni di denominazione interne.
I prefissi come \texttt{v\_}, \texttt{f\_} o \texttt{param} identificano inequivocabilmente le variabili
locali, i campi di classe o i parametri formali dei metodi.
Questa classificazione rigorosa determina la corretta strategia di risoluzione AST successiva.
Di seguito il blocco Python \texttt{\_classify\_entity\_kind} che implementa questa logica essenziale:

\begin{lstlisting}[language=Python]
def _classify_entity_kind(prolog_id: str) -> str:
    # Classificazione basata sull'ispezione del prefisso dell'identificatore Prolog
    if prolog_id.startswith("v_"):
        return "variable"
    elif prolog_id.startswith("param") or prolog_id.startswith("p_"):
        return "parameter"
    elif prolog_id.startswith("f_"):
        return "field"
    elif "expr" in prolog_id or "literal" in prolog_id:
        return "expression"
    return "unknown"
\end{lstlisting}

\subsubsection{La Risoluzione dei Tipi Tramite Tree-sitter AST}
Per recuperare il tipo Java di ciascuna entità, il sistema usa la libreria Tree-sitter come parser
incrementale, interrogando direttamente l'AST senza dover compilare l'intero progetto. Una funzione di
smistamento (\texttt{\_resolve\_type\_at\_line}) instrada la richiesta al risolutore specializzato in base alla
categoria calcolata in precedenza: variabili locali, parametri, campi o espressioni. I quattro risolutori
condividono l'idea di fondo ma differiscono nello scope di ricerca; ne vediamo il caso rappresentativo, quello
delle variabili locali.

Per le \textbf{entità dichiarate}, variabili locali, parametri formali e campi, la strategia è la stessa:
raggiungere il nodo di dichiarazione nell'AST e leggerne il tipo. I tre casi differiscono solo nello
\emph{scope} di ricerca:
\begin{itemize}
    \item le \textbf{variabili locali} si cercano nel solo metodo che contiene il crash, con una DFS che individua il nodo; 
    poiché l'AST separa il tipo dal dichiaratore, si raccoglie il nodo del
    tipo (nelle sue varie forme: \texttt{type\_identifier}, \texttt{generic\_type}, \texttt{array\_type}\dots) e
    il \texttt{variable\_declarator} il cui nome coincide con la variabile cercata;
    \item i \textbf{parametri formali} sono più immediati: l'AST espone nome e tipo come campi (\texttt{name} e
    \texttt{type}) del nodo \texttt{formal\_parameter}, accessibili con \texttt{child\_by\_field\_name}, senza
    bisogno della DFS;
    \item i \textbf{campi di istanza/classe} usano la stessa logica delle variabili locali, ma estesa all'intero
    albero anziché al singolo metodo, perché le dichiarazioni di campo risiedono a livello di classe.
\end{itemize}

Per le \textbf{espressioni composte} (\texttt{entity\_kind == "expression"}), la risoluzione esatta del tipo restituito 
non è possibile tramite una semplice ricerca di dichiarazione, poiché dipende dalla semantica dei metodi invocati. 
Pertanto, il sistema applica un'euristica mirata tramite pattern matching per riconoscere le operazioni di 
\textit{cast esplicito}. L'espressione regolare è stata studiata per intercettare non solo tipi semplici, 
ma anche strutture sintatticamente più ricche, includendo costrutti Java avanzati (come tipi generici o array).

\begin{lstlisting}[language=Python]
def _infer_type_from_expression(expr: str) -> str:
    """Inferisce il tipo di un'espressione tramite pattern matching."""
    # Pattern: (Tipo)espressione -> il tipo dichiarato nel cast è assunto come tipo
    cast_match = re.match(r"\((\w[\w<>\[\],\s]*)\)", expr)
    if cast_match:
        return cast_match.group(1).strip()
    return ""
\end{lstlisting}

Ad esempio, in un altro bug del dataset, data l'espressione \texttt{(XYItemRenderer)this.ren\\derers.get(i)}, 
il pattern isola ed estrae la stringa \texttt{XYItemRenderer}.

\textbf{Meccanismo di fallback testuale:} qualora l'analisi tramite Tree-sitter non fosse disponibile o 
fallisse la ricerca nell'albero, il sistema possiede un meccanismo di fallback. 
Questa procedura compie una scansione testuale all'indietro (\textit{backward search}) limitata alle 
50 righe antecedenti la riga del crash, una finestra empiricamente sufficiente a coprire la 
lunghezza tipica dei metodi analizzati nel benchmark Defects4J. L'espressione regolare utilizzata 
per questo compito supporta identificatori complessi, tenendo conto dell'eventuale modificatore 
\texttt{final}, tipi parametrici, array e path di package.

\begin{lstlisting}[language=Python]
def _resolve_type_regex(lines, line_no, java_name, entity_kind) -> str:
    """Fallback regex per la risoluzione del tipo in backward search."""
    # Cattura tipi complessi ignorando l'eventuale modificatore 'final'
    var_re = re.compile(
        rf"(?:final\s+)?([\w<>\[\],.?]+)\s+{re.escape(java_name)}\b"
    )
    search_start = max(0, line_no - 50)
    for i in range(line_no - 1, search_start - 1, -1):
        m = var_re.search(lines[i])
        if m:
            candidate = m.group(1).strip()
            # Successivamente il sistema esclude parole chiave riservate 
            # (es. 'return') che potrebbero attivare falsi positivi
            return candidate
    return ""
\end{lstlisting}

L'approccio testuale è relegato a \textit{fallback} perché, a differenza dell'analisi strutturale AST, le regex
presentano limiti intrinseci:
\begin{itemize}
    \item \textbf{Falsi positivi:} possono intercettare codice commentato, letterali stringa o parole chiave che
    mimano sintatticamente una dichiarazione;
    \item \textbf{Mancanza di scoping:} la scansione all'indietro è ``cieca'' rispetto all'annidamento dei blocchi
    e ai casi di \textit{shadowing}, mentre l'AST mantiene piena consapevolezza dello scope;
    \item \textbf{Fragilità sintattica:} ritorni a capo arbitrari, generici annidati, array e annotazioni rendono
    le espressioni regolari complesse e fragili.
\end{itemize}
La procedura a ritroso funge quindi da rete di sicurezza per i fallimenti puntuali dell'analisi strutturale.

\subsubsection{La Classificazione delle NPE}

Prima della classificazione, un parser a espressioni regolari (in \texttt{prolog\_grounding.py}) disassembla il
formato testuale di \texttt{root\_cause.txt}, già illustrato in precedenza, in record grezzi, uno per ciascun
anello della catena causale, aggregati per sito di crash. È su questi record che opera la risoluzione dei tipi
descritta sopra, producendo le entità Java tipizzate.

A partire dall'entità nulla e dalla lista delle cause, il sistema classifica poi ogni NPE in una di sei
categorie, secondo una tassonomia derivata dalle regole di \texttt{npe-rules.pl}: ispeziona le regole Prolog che
hanno generato le cause e assegna la categoria corrispondente. La mappatura è riassunta nella tabella seguente
(per Csv-4 la categoria risultante è \texttt{UNINITIALIZED\_FIELD}):

\begin{table}[h]
\centering
\begin{tabular}{lp{4cm}p{5cm}}
\toprule
\textbf{Categoria Python} & \textbf{Regola Prolog} & \textbf{Semantica} \\
\midrule
NULL\_RETURN & \texttt{is\_null\_return} & Il metodo ha percorsi di ritorno null e non-null \\
NULL\_ARGUMENT & \texttt{null\_arg\_passed} & Un argomento null viene passato a un metodo \\
UNINITIALIZED\_FIELD & \texttt{null\_ref} (tipo field) & Un campo non è stato correttamente inizializzato \\
NULL\_ARRAY\_ACCESS & \texttt{null\_ref} (su indicizzazione) & Un elemento specifico di array/collection è null \\
NULL\_DEREFERENCE & \texttt{null\_ref} generico & Dereferenziazione diretta e immediata di variabile nulla \\
NULL\_LITERAL\_ASSIGN & \texttt{val} con literal in cause & Assegnazione esplicita e diretta al valore null \\
\bottomrule
\end{tabular}
\end{table}

\subsubsection{La Struttura Dati GroundedNPE}

Il risultato del Semantic Grounding è l'oggetto \texttt{GroundedNPE}, che struttura gerarchicamente gli strati
informativi dell'inferenza. Questa gerarchia risiede in \texttt{shared/models.py} e usa le \texttt{dataclass} di
Python per contratti tipizzati:

\begin{lstlisting}[language=Python,breaklines=false]
from dataclasses import dataclass
from typing import List

@dataclass
class GroundedEntity:
    """Un'entità Prolog risolta nella sua identità Java reale."""
    prolog_id: str      # Es. "f_header_map_69" (ID opaco Prolog)
    java_name: str      # Es. "headerMap" (nome Java leggibile)
    java_type: str      # Es. "Map<String,Integer>" (tipo risolto via AST)
    entity_kind: str    # Es. "field" (ruolo semantico)
    source_line: int    # Es. 288 (riga sorgente, 1-indexed)
\end{lstlisting}
\begin{lstlisting}[language=Python]
@dataclass
class GroundedCause:
    """Una singola causa nella catena causale, con contesto Java risolto."""
    entity: GroundedEntity
    explanation: str    # Spiegazione testuale, es. "Explicitly assigned `null`"

@dataclass
class GroundedNPE:
    """Diagnosi NPE completamente grounded con classificazione e direttiva."""
    crash_line: int               # Riga in cui si verifica il crash
    class_id: str                 # ID identificativo della classe
    null_entity: GroundedEntity   # L'entità nulla nel sito del crash
    causes: List[GroundedCause]   # Catena ordinata delle cause radice
    category: NPECategory         # Classificazione tassonomica del bug
    repair_directive: str         # Direttiva generata per la riparazione
\end{lstlisting}

Questa gerarchia confluisce nel campo \texttt{grounded\_chains} (una \texttt{List[GroundedNPE]}) dell'oggetto
\texttt{FaultReport}, prodotto finale della Fase 1 e payload di input per la generazione dei prompt LLM.


\subsubsection{Direttive di Riparazione (Rule-to-Prompt)}

Sulla base della classificazione, il sistema genera una direttiva di riparazione specifica per categoria. Il
dizionario \texttt{\_REPAIR\_DIRECTIVES} implementa il meccanismo ``Rule-to-Prompt'' tramite template parametrici:
\begin{lstlisting}[language=Python, showstringspaces=false, breaklines=true]
_REPAIR_DIRECTIVES: Dict[NPECategory, str] = {
    NPECategory.NULL_RETURN: (
        "The method {cause_expr} can return null. "
        "Add a null-check on its return value before dereferencing, "
        "OR modify the method to guarantee a non-null return value "
        "(e.g., return an empty collection or a default object)."
    ),
    NPECategory.NULL_ARGUMENT: (
        "The parameter `{null_name}` can be null when passed to this method. "
        "Add an early null guard at the top of the method "
        "(e.g., `if ({null_name} == null) return ...;`), "
        "OR ensure the caller never passes null."
    ),
    NPECategory.UNINITIALIZED_FIELD: (
        "The field `{null_name}` is not initialized (or explicitly set to null). "
        "Add initialization in the constructor or a lazy-init null-check before use. "
        "Consider what the correct default value should be."
    ),
    NPECategory.NULL_ARRAY_ACCESS: (
        "The element retrieved from `{null_name}` is null. "
        "Add a null-check on the retrieved element before dereferencing it, "
        "OR ensure the array/collection never stores null at that position."
    ),
    NPECategory.NULL_DEREFERENCE: (
        "The variable `{null_name}` is null when dereferenced at line {crash_line}. "
        "Add a null guard before the dereference, "
        "OR trace back why it is never assigned a non-null value."
    ),
    NPECategory.NULL_LITERAL_ASSIGN: (
        "The variable `{null_name}` is explicitly assigned `null` at line {assign_line} "
        "and later dereferenced at line {crash_line}. "
        "Change the null assignment to a safe default, or add a null-check "
        "before the dereference."
    )
}
\end{lstlisting}

Le direttive sono parametriche rispetto alle informazioni grounded: segnaposto come \texttt{\{null\_name\}}
vengono sostituiti dinamicamente con il nome Java reale dell'entità nulla. La parametrizzazione trasforma così
un'analisi statica astratta in un'istruzione imperativa, direttamente comprensibile dal modello generativo.
La direttiva così istanziata viene salvata nel campo \texttt{repair\_directive} del rispettivo \texttt{GroundedNPE}
(e quindi in \texttt{FaultReport.grounded\_chains}): non è ancora il prompt completo, ma il frammento operativo
che la fase di riparazione inserirà nel prompt inviato all'LLM.

\subsubsection{Perché Questo Aiuta l'LLM}

Il confronto tra l'output grezzo di LogicFL e quello arricchito dal grounding illustra il vantaggio di questa
architettura.

\textbf{Prima (Grezzo - Output di LogicFL)}
\begin{lstlisting}[language={}, basicstyle=\ttfamily\small, keywordstyle={}, stringstyle={}, commentstyle={}, showstringspaces=false]
NPE at line(csvparser_1, 288) / Null Expression - f_header_map_69[headerMap]
\end{lstlisting}

\textbf{Dopo (Strutturato - Input per l'LLM)}
\begin{lstlisting}[language={}, basicstyle=\ttfamily\small, keywordstyle={}, stringstyle={}, commentstyle={}, showstringspaces=false]
Categoria: UNINITIALIZED_FIELD
Entita: headerMap (Tipo: Map<String, Integer>) alla riga 288
Direttiva: "The field `headerMap` is not initialized (or explicitly set 
to null). Add initialization in the constructor or a lazy-init 
null-check before use. Consider what the correct default value should be."
\end{lstlisting}

Ricevendo questo input mirato, l'agente LLM di HybridRepair è in grado di produrre, al suo secondo tentativo, una patch che supera la suite di test di Csv-4, introducendo un ternary operator che gestisce esplicitamente il caso non inizializzato:
\begin{lstlisting}[language=Java]
// Patch generata da HybridRepair per Csv-4
public Map<String, Integer> getHeaderMap() {
    return this.headerMap == null ? null : new LinkedHashMap<>(this.headerMap);
}
\end{lstlisting}

La differenza qualitativa si articola su quattro dimensioni:
\begin{itemize}
    \item \textbf{Risoluzione dei tipi:} l'output grounded include il tipo Java (\texttt{Map<String, Integer>}),
    assente nel testo grezzo; il modello può ragionare sul costruttore di classe più appropriato.
    \item \textbf{Classificazione semantica:} la categoria \texttt{UNINITIALIZED\_FIELD} focalizza l'attenzione 
    sul ciclo di vita del campo all'interno della classe, suggerendo una soluzione mirata.
    \item \textbf{Direttiva operativa:} la direttiva traduce l'analisi logica in un'istruzione d'azione concreta,
    guidando l'LLM ad implementare un null-check piuttosto che modifiche arbitrarie, garantendo una \textit{constrained generation}.
    \item \textbf{Eliminazione dell'ambiguità simbolica:} gli identificatori opachi (\texttt{f\_header\_map\_69}) sono
    sostituiti dalle controparti Java, evitando che il modello li confonda con nomi di variabili reali.
\end{itemize}

In sintesi, mentre i modelli puramente statistici operano \textit{a posteriori} sul testo sorgente, HybridRepair
combina la certezza deduttiva di Prolog con la flessibilità generativa del modello: il Semantic Grounding è il
ponte tra logica e probabilità.

\subsubsection{Robustezza e Degradazione Controllata}

Il modulo \texttt{prolog\_grounding.py} segue una filosofia di \textit{graceful degradation}: ogni fase può
fallire senza compromettere il sistema. Se Tree-sitter è irreperibile, si ricade sul fallback a regex; se il
tipo resta irrisolto, il campo è omesso e la riparazione procede con una direttiva parzialmente istanziata; se
\texttt{root\_cause.txt} non è reperibile, si usano le catene causali grezze. Questo evita \textit{single points of failure}.

\subsection{Integrazione nella Pipeline Python: Il Modulo FaultOracle}

L'integrazione del processo diagnostico nella pipeline Python è orchestrata dal modulo \texttt{fault\_oracle/},
che media tra il motore Prolog e le fasi di riparazione neurale. Il modulo \texttt{logicfl\_runner.py} gestisce
il ciclo di vita dell'analisi LogicFL, riusando i risultati già disponibili per evitare riesecuzioni ridondanti.
Terminato il motore Prolog, \texttt{logicfl\_parser.py} legge i file generati e ne coordina l'arricchimento
semantico (\textit{Semantic Grounding}), consolidando localizzazioni, catene causali grezze e analisi AST nel
report finale:

\begin{lstlisting}[language=Python]
def parse_logicfl_output(bug_dir: Path) -> FaultReport:
    # 1. Estrazione delle localizzazioni con contesto AST
    locations = _parse_fault_locations(result_dir / "fault_locs.txt", source_roots)

    # 2. Parsing delle catene causali grezze prodotte da Prolog
    root_cause_text, causal_chains = _parse_root_cause(result_dir / "root_cause.txt")

    # 3. Risoluzione e arricchimento tramite Semantic Grounding
    grounded_chains = ground_causal_chains(
        root_cause_text, source_file=locations[0].file_path, ...
    )

    # Generazione e ritorno del report diagnostico unificato
    return FaultReport(
        bug_id=bug_id,
        locations=locations,
        root_cause_text=root_cause_text,
        causal_chains=causal_chains,
        grounded_chains=grounded_chains
    )
\end{lstlisting}

Il \texttt{FaultReport} finale è l'interfaccia strutturata tra la diagnostica e le fasi successive della
pipeline: mettendo a disposizione righe sospette, spiegazione dell'errore e analisi causale, garantisce che la
generazione della patch disponga di tutto il contesto necessario.
